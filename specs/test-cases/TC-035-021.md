# pr.commented rule in plugin.yaml triggers spec inbox end-to-end

## Preconditions

- `gitcheck` CLI is installed with the spec-pipeline plugin
- Plugin.yaml contains:
  ```yaml
  - type: pr.commented
    labels: ["spec:*"]
    action: "/spec inbox"
  ```
- `GITHUB_TOKEN` is set
- Repository `owner/test-repo` has open PR #70 with label `spec:requirements`
- A reviewer posts a comment: "Please add error handling for the edge case in section 2"

## Steps

1. **Action:** Start `gitcheck listen --poll --repo owner/test-repo --interval 10s` and wait for the poll cycle to detect the new comment.
   **Expected:** Listener emits a `pr.commented` event for PR #70.

2. **Action:** EventRouter matches the event to the `pr.commented` rule.
   **Expected:** Action `/spec inbox` is resolved and dispatched to the agent session.

3. **Action:** Agent session starts with the `/spec inbox` prompt and the comment context available as variables.
   **Expected:** Claude Code receives the spec inbox command. The comment body and author are available for the agent to use when addressing the feedback.

4. **Action:** Agent processes the comment, makes changes, and pushes a commit.
   **Expected:** A new commit appears on the PR branch addressing the reviewer's feedback.

## Postconditions

- Full loop: comment -> event -> plugin match -> agent session -> code change -> commit
- The reviewer's feedback is addressed without manual intervention
